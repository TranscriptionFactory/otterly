## Bug #1: File-modified-on-disk race condition (autosave conflict) — P0

**Status:** Implemented

### Root Cause Analysis

The `WatcherService` used a **counter-based suppression** mechanism to prevent the app's own writes from being detected as "external modifications." Each `suppress_next()` call incremented a counter by 1, and each `is_suppressed()` call consumed one token by decrementing.

**The problem:** macOS FSEvents (via the Rust `notify` crate) frequently emits **multiple `Modify` events** for a single file write. The counter started at 1, the first event consumed the token, and subsequent events slipped through — triggering the spurious "Note has been modified externally" toast.

Additionally, if a user clicked "Keep my changes," the mtime guard was set to 0, disabling conflict detection entirely. A subsequent autosave could then silently overwrite a genuine external edit.

### Fix

Replaced counter-based suppression with **timestamp-based suppression** in `WatcherService`:

- `suppress_next(path)` now stores `Date.now()` instead of incrementing a counter.
- `is_suppressed(path)` checks whether the current time is within the 10-second suppression window. It returns `true` for **all events** within the window, regardless of how many arrive.
- No timeout cleanup needed — stale entries are lazily evicted when checked.

This eliminates the race condition because any number of FSEvents from a single write are suppressed, while genuine external modifications after the window will still trigger the conflict toast.

### Files Changed

- `src/lib/features/watcher/application/watcher_service.ts` — core fix
- `tests/unit/services/watcher_service.test.ts` — updated tests for new semantics

## Bug #2: LaTeX equations broken — P0

**Status:** Implemented

### Root Cause Analysis

Investigation revealed three distinct issues causing the "math not working" regression:

1. **No input rules for math syntax.** The math plugin defined ProseMirror node schemas and NodeViews for rendering, but provided no mechanism for users to _type_ math in the editor. Only the `/math` slash command existed (block math only). When a user typed `$E=mc^2$`, it remained as literal text — never converted to a `math_inline` node.

2. **Line break preprocessor corrupted math blocks.** The `prepare_markdown_line_breaks_for_visual_editor` function tracked code fences (`` ` `` and `~`) to skip preprocessing inside code blocks, but did not recognize `$$...$$` math block fences. Lines inside math blocks ending with a single `\` (common in LaTeX, e.g., `\frac{1}{2}\`) were incorrectly converted to `<br />`, corrupting the LaTeX source before it reached remark-math.

3. **MathBlockNodeView didn't propagate updates.** The `update()` method reassigned `this.node` but never re-mounted the Svelte component with the new node reference. This meant undo/redo or external attribute changes to a math_block node were silently ignored by the rendered component.

### Fix

**Input rules (math_plugin.ts):**

- Added `math_inline_input_plugin`: an `appendTransaction` plugin that detects `$content$` patterns in text and converts them to `math_inline` ProseMirror nodes. Uses regex `/(?:^|[^$\\])\$([^$\n]+)\$$/` to avoid false positives on `$$` (block delimiter) or `\$` (escaped dollar).
- Added `math_block_input_rule`: a ProseMirror `InputRule` that converts `$$` typed on an empty line into a `math_block` node with the interactive editor.

**Preprocessor protection (markdown_line_breaks.ts):**

- Added `is_math_fence_line()` function to detect `$$` fence lines.
- Added `in_math_block` state tracking to both `normalize_markdown_line_breaks` and `prepare_markdown_line_breaks_for_visual_editor`, mirroring the existing `fence_state` pattern for code blocks.

**NodeView update propagation (math_plugin.ts):**

- `MathBlockNodeView.update()` now compares `updated.attrs["value"]` with the current node's value. On change, it unmounts and remounts the Svelte component with the new node reference.

- `tests/unit/domain/markdown_line_breaks.test.ts` — 5 new math block tests
